## 框架设计

### 1. Token 管理
- **方案**: fixture + 自动刷新
- **说明**: 登录一次获取 Token，自动检测过期并刷新

### 2. 测试数据管理
- **方案**: YAML 文件
- **位置**: `data/` 目录
- **格式**: 按模块分文件，如 `user_login.yaml`

### 3. 断言策略
- **方案**: 校验具体字段值
- **校验内容**:
  - `code` 字段值
  - `message` 字段值
  - `data` 内关键字段值

### 4. 报告输出
- **方案**: allure
- **输出**: `reports/allure-results/`
- **查看**: `allure serve reports/allure-results`

## 测试用例设计方法

### 核心原则：基于文档，分层覆盖

| 方法 | 说明 | 示例 |
|------|------|------|
| **等价类划分** | 把输入分成有效/无效等价类 | 手机号：有效11位、无效10位/12位/空 |
| **边界值分析** | 测试边界及边界附近值 | 分页：page=1, page=最大页, page=0, page=负数 |
| **场景法** | 按业务流程设计 | 添加→查询→更新→删除 完整流程 |
| **错误推测** | 基于经验预判易错点 | 特殊字符、SQL注入、超长字符串 |
| **正交实验** | 多字段组合时用，减少用例数 | 3字段各2值→8种组合而非27种 |
| **状态迁移** | 有状态变化时 | 订单：待支付→已支付→已发货→已完成 |
| **参数校验矩阵** | 每个必填字段 × 缺失/空值/null/格式错误 | 必填字段全覆盖 |

### 参数校验矩阵模板

对于每个必填字段，至少覆盖：
1. **正常值** - 符合要求的有效值
2. **缺少字段** - 请求体中不包含该字段
3. **字段为空** - 字段存在但值为空字符串 `""`
4. **字段为null** - 字段存在但值为 `null`
5. **格式错误** - 类型不匹配、长度超限、非法字符

### 关键步骤
1. 先读接口文档，提取所有约束（必填、类型、长度、范围）
2. 列出所有字段的校验规则
3. 用矩阵法覆盖每个字段的各种异常情况
4. 补充业务场景（正常流程、异常流程）
5. 考虑安全/性能相关（并发、权限、越权）

## 经验教训（2026-04-22 总结）

### 必须遵守的原则

| 原则 | 说明 | 优先级 |
|------|------|--------|
| **先调接口再写用例** | 用 curl/postman 验证接口实际行为，再写预期值 | ⭐⭐⭐ |
| **类型要兼容** | 接口返回的 code 可能是 string 或 number，断言时要兼容 | ⭐⭐⭐ |
| **代码要适配数据** | YAML 加用例时，代码要能自动识别（参数化） | ⭐⭐⭐ |
| **数据要清理** | 无论成功失败，只要有 data.id 就要调用删除接口 | ⭐⭐⭐ |
| **命名要清晰** | 用 pytest_generate_tests 生成可读用例名 | ⭐⭐ |
| **Token 要可控** | 测试无Token/错误Token场景时，必须禁用自动Token | ⭐⭐⭐ |

### 常见错误

1. **凭经验写预期值** → 必须先调接口确认实际返回值
2. **假设接口会校验参数** → 实际可能不校验，要验证
3. **手动写测试方法** → 应该用参数化自动生成
4. **只在成功用例里清理数据** → 所有用例都要检查并清理
5. **忽略类型转换** → string/number 要统一比较
